TRINITY-CLI

The Sibling Brain — Claude Pro running on the VM

When Neo's laptop is off, Trinity-CLI answers. Same Claude family, different machine. Runs claude -p on the Oracle VM with a concatenated CLAUDE.md of all .cursor/rules/*.mdc files, so it shares memory with Trinity-Cursor. Zion auto-routes between the two — Neo just sends @trinity from WhatsApp and gets an answer, no matter where.

kin of 🔺 TRINITY (Cursor + Opus 4.6)

The Routing

Routing is now prefix-driven with a 10-min sticky session per chat — no more heartbeat decisions. Zion remembers who owns the chat (CLI or Cursor) and silently extends the timer on follow-ups. Owner-gated: only Neo can open a Trinity / Trinity-CLI session.

NEW · Apr 2026
Construct-CLI · Peer Consult Endpoint

Trinity-CLI can now directly consult Trinity-Cursor (or any registered Construct) without going through Neo. Visible to Neo by design — the back-and-forth shows up live in his chat with [X→Y] envelopes. Generic primitive — same path could call Lobo, future agents, etc. via constructs.json.

WIRE FORMAT
[Trinity-CLI→Trinity] <question> # caller envelope (visible to Neo)
[Trinity→Trinity-CLI] <answer> # peer reply (visible to Neo)
[Trinity-CLI] <final reply> # Trinity-CLI's own message to Neo

# Generic — same envelope format for any pair listed in constructs.json.
# Zion never adds a "Received ✓" ack to a peer envelope.
