🧠 The Enrich — How Cortex Gets Smarter
The Problem
A team member asks: "@neo what is KYC Gateway?"

Cortex searches 20GB of documents. Finds some raw chunks — a few sentences from a design doc, a table header from an Excel. Sends it to GPT-4.1.

GPT gives an answer, but it's thin and fragmented — bits and pieces from random docs. Not a real explanation. The team member walks away unimpressed.
Every question is logged
Cortex logs every question in the question_log table — who asked, when, what the answer was, and how confident the system was.
Trinity steps in — "ENRICH ON"
Neo says "ENRICH ON". Trinity (Opus 4.6) wakes up and does this:

1. Reads question_log — finds questions that got weak answers
2. Identifies the weak topics — "KYC Gateway got asked 4 times but the answer was always thin"
3. Goes to the actual source documents — reads the full HLD, the design doc, the architecture PowerPoint, not just tiny chunks
4. Writes a deep, flowing explanation — like a senior engineer would explain it to a new joiner
5. Saves it back into Cortex's database as an enriched document
Next time the same question is asked
Now when someone asks "what is KYC Gateway?" — Cortex finds the enriched doc first (it's scored higher than raw chunks).

GPT-4.1 gets a complete, well-written explanation instead of fragmented pieces. The answer is now confident, detailed, and accurate.

The team member says: "wow, this actually knows our stuff."
The learning loop
Weak answer → logged → Trinity reads source docs deeply → writes enrichment → strong answer next time

Every ENRICH run makes Cortex smarter. It's not retraining the AI — it's giving the AI better source material to work with.

20 enrichments written so far across 3 accounts. Each one transforms a "meh" answer into a "that's exactly what I needed" answer.

🔮 ChromaDB — Vector Search (Semantic)
What it does
ChromaDB stores 349,000 text chunks from 20GB of documents — but not as plain text. Each chunk is converted into a 384-dimension number vector (an "embedding").

These vectors capture meaning, not words. So when you search, it finds chunks that are about the same thing — even if they use completely different words.
Example: Why "car" finds "automobile"
You search: "car"

📄 "The automobile engine..." 0.92 match
📄 "Vehicle maintenance..." 0.89 match
📄 "Sedan and SUV models..." 0.85 match
📄 "Kitchen appliance guide..." 0.12 match

None of these documents contain the word "car" — but the AI understands they're about the same thing. A keyword search would find zero results.
Real Cortex example
Someone asks: "@neo what is the KYC verification flow?"

ChromaDB finds chunks mentioning "identity check", "customer onboarding", "document validation" — even though those docs never use the exact phrase "KYC verification flow."

Model: bge-small-en-v1.5 · 384 dimensions · ONNX runtime · runs locally (no cloud API needed)

🔍 SQLite FTS5 — Keyword Search
What it does
FTS5 is Full-Text Search — the old-school way. It indexes every word in every document and finds exact matches.

Fast, precise, and catches things that semantic search misses — like product codes, server names, version numbers, and abbreviations.
Example: "KYC-4021" — only keywords work here
You search: "KYC-4021"

📄 "...KYC-4021 handles tier-2 banks..." ✓ exact hit
📄 "...deploy KYC-4021 to staging..." ✓ exact hit
📄 "...identity verification..." ✗ no match

Semantic search would miss "KYC-4021" entirely — it doesn't understand product codes. FTS5 finds it instantly.
Why BOTH are needed (Hybrid Search)
ChromaDB = "what does the question mean?" (car → automobile ✓)
FTS5 = "what exact words appear?" (KYC-4021 → exact hit ✓)

Cortex runs both searches in parallel, merges results, and sends the best chunks to GPT-4.1. This is called Hybrid Search — you get the best of both worlds.

Also includes enriched docs (Trinity-written deep explanations) scored higher than raw chunks.
